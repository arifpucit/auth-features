Security Vulnerability Lab Feature 01 / 05 · Email Verification

Prove the email is real before the account is.

On sign-up the app mints a one-time token, mails a link through SendGrid's HTTPS API, and only activates the account once that link is clicked — before the token expires.

✓Verify

What happens

The verification flow

The path a brand-new account takes from sign-up to verified — at the abstract level.

  1. 1

    User submits the sign-up form

    They type an email address and choose to create an account. The account exists but is marked unverified.

  2. 2

    The app mints a token

    A unique, hard-to-guess verification token is generated and stored against the account with an expiry timestamp.

    ⏱ default lifetime: 1 hour
  3. 3

    SendGrid mails the link

    The app calls SendGrid's HTTPS API — not SMTP — to send an email containing APP_BASE_URL + the token. (HTTPS is used because hosts like Render block outbound SMTP.)

  4. 4

    User clicks the link

    Opening the link in their browser hits the verify endpoint and hands back the token from the email.

  5. 5

    The endpoint checks the token

    It looks the token up, confirms it exists and has not expired. An expired or unknown token is rejected — the user can request a fresh one.

  6. ✓

    Account marked verified

    The token is consumed and the account flips to verified. The email address is now proven to belong to the user.

    ● status: verified
What you do

Run it yourself

Three values turn email on. Get them once from SendGrid, then point them at wherever the app runs.

  1. Create a free SendGrid account.
  2. Verify a Single Sender — prove you own the From address (or verify a whole domain).
  3. In SendGrid: Settings → API Keys → Create API Key with Mail Send permission.
  4. Drop the three values into the right place for your environment ↓
.envcopied from .env.example
# your real .env is git-ignored — never commit it
SENDGRID_API_KEY=SG.your-sendgrid-api-key
SENDGRID_FROM=your-verified@example.com
APP_BASE_URL=http://localhost:3001
Service → Environmentno .env uploaded
# add each as a key / value in the dashboard
SENDGRID_API_KEY → SG.your-sendgrid-api-key
SENDGRID_FROM → your-verified@example.com
APP_BASE_URL → https://your-app.onrender.com
⚠️APP_BASE_URL builds the link inside the email, so it must match where the app actually runs (no trailing slash). Leave all three unset and sign-up simply shows a friendly “email not configured” message.

Security Vulnerability Lab Feature 02 / 05 · Continue with Google

Let Google vouch for who's logging in.

The user authenticates on Google's own page, then the app trusts Google's answer — the public Client ID rides along in the browser, while the private Client Secret never leaves the server.

What happens

The OAuth sign-in flow

The authorization-code dance between the app, the browser, and Google — and where each key is used.

  1. 1

    User clicks “Continue with Google”

    The app redirects the browser to Google, putting the public Client ID and the registered redirect_uri in the URL. No secret here — this part is openly visible in the browser.

    🌐 public Client ID — identifies the app
  2. 2

    User signs in & consents at Google

    They authenticate on Google's own page and approve access. The app never sees their Google password.

  3. 3

    Google redirects back with a code

    The browser is sent to your exact redirect_uri carrying a short-lived, one-time authorization code.

  4. 4

    Server swaps the code for tokens

    Server-to-server, the app sends the code plus the private Client Secret to Google. The secret proves the request is really from your app — it never touches the browser.

    🔒 private Client Secret — proves it's your app
  5. 5

    Google returns the user's identity

    Google hands back the verified profile (email, name). The app matches or creates a local account.

  6. ✓

    User is logged in

    A local session starts — all without the app ever handling a Google password.

    ● session created
What you do

Run it yourself

Get one public key, one private key, and one exact-match redirect URL from Google.

  1. In Google Cloud Console → Google Auth Platform → Clients, create an OAuth client of type Web application.
  2. Copy the Client ID (public) and Client Secret (private).
  3. Register an Authorized redirect URI — it must match exactly: scheme, host, port, path, no trailing slash.
  4. Put the three values where the app runs ↓
.envcopied from .env.example
# public id is fine to expose; never commit the secret
GOOGLE_CLIENT_ID=your-id.apps.googleusercontent.com
GOOGLE_CLIENT_SECRET=your-client-secret
GOOGLE_REDIRECT_URI=http://localhost:3001/auth/google/callback
Service → Environmentno .env uploaded
# also add this exact URL in the Google console
GOOGLE_CLIENT_ID → your-id.apps.googleusercontent.com
GOOGLE_CLIENT_SECRET → your-client-secret
GOOGLE_REDIRECT_URI → https://your-app.onrender.com/auth/google/callback
⚠️When you deploy, change GOOGLE_REDIRECT_URI and add that same URL to the allowed list back in the Google console. If the two don't match, Google rejects login with redirect_uri_mismatch.

Security Vulnerability Lab Feature 03 / 05 · Cloudflare CAPTCHA

A human gate before the password check.

Turnstile drops a human-check widget onto the login form using a public Site Key. On submit, the server confirms the result with Cloudflare using the private Secret Key — before it ever looks at a password.

✓
Success!Verified you're human
Turnstile
What happens

The CAPTCHA flow

Where the widget sits in the login path, and which key does what.

  1. 1

    Login page renders the widget

    Using the public Site Key, Turnstile draws its widget on the login form (often passing invisibly for real users).

    🌐 public Site Key — in the page
  2. 2

    User passes the human check

    Turnstile runs its checks and hands the browser a one-time response token.

  3. 3

    User submits the login form

    The form posts the username and password together with that Turnstile token.

  4. 4

    Server verifies with Cloudflare first

    Before any password check, the server sends the token plus the private Secret Key to Cloudflare's siteverify endpoint.

    🔒 Secret Key — server only
  5. 5

    Cloudflare answers

    Valid → continue to the password check. Invalid → reject the attempt before credentials are ever read.

  6. ✓

    Then — and only then — the password is checked

    The human gate sits strictly in front of authentication.

    ● gate cleared
What you do

Run it yourself

Free, no domain needed. Grab a public Site Key and a private Secret Key.

  1. Go to dash.cloudflare.com → Turnstile → Add widget.
  2. Set Hostname to localhost (add your real domain for production); choose Managed mode.
  3. Copy the Site Key (public) and Secret Key (private).
  4. Put both where the app runs ↓
.envhostname: localhost
# site key is public; never commit the secret
TURNSTILE_SITE_KEY=your-turnstile-site-key
TURNSTILE_SECRET_KEY=your-turnstile-secret-key
Service → Environmentadd onrender.com host
# add your onrender.com hostname in the Turnstile widget
TURNSTILE_SITE_KEY → your-turnstile-site-key
TURNSTILE_SECRET_KEY → your-turnstile-secret-key
✅Optional & safe: leave both unset and the widget simply doesn't appear — login works exactly as before (graceful degrade).
🛟Fails open: if the keys are set but Cloudflare can't be reached, login is allowed through, so a Cloudflare outage can't lock everyone out.

Security Vulnerability Lab Feature 04 / 05 · QR Login

Scan from a signed-in device, skip the password.

An unauthenticated browser shows a QR. A device that's already signed in scans and approves it — and the first browser logs straight in. No email, no Google, no extra dependency.

Scan to log in ⏱ valid 120s · polls every 2s
What happens

The QR login flow

How a waiting browser and an already-trusted device hand off a session.

  1. 1

    Browser shows a QR on /login

    An unauthenticated browser displays a QR encoding a short-lived login request, then quietly polls the server — “approved yet?” — on an interval.

    ⏱ QR valid 120s · poll every 2s
  2. 2

    An already-signed-in device scans it

    A phone — or a second browser that's already logged in — opens the link the QR encodes.

  3. 3

    The signed-in device approves

    The user taps approve. The server marks that specific login request as approved.

  4. 4

    The waiting browser sees the approval

    On its next poll, the first browser picks up the new “approved” status.

  5. ✓

    The first browser logs in

    A session starts on the waiting browser — no password typed anywhere.

    ● session created
What you do

Run it yourself

The easiest of the four — it works on a fresh clone. You only adjust reachability for a real scan.

  1. Nothing required — works on a fresh clone with the defaults.
  2. For a real phone scan, point APP_BASE_URL at an address the scanning device can reach — your LAN IP or your Render URL, not localhost.
  3. On one machine, “scan” by opening the QR's link in a second, already-logged-in browser.
  4. Optionally tune the timing values ↓
.envall optional
# defaults shown — use a LAN IP so a phone can reach it
APP_BASE_URL=http://192.168.1.20:3001
QR_LOGIN_TTL_SECONDS=120
QR_LOGIN_POLL_INTERVAL_SECONDS=2
Service → Environmentpublic URL reachable
# the public Render URL is reachable by any phone
APP_BASE_URL → https://your-app.onrender.com
QR_LOGIN_TTL_SECONDS → 120
QR_LOGIN_POLL_INTERVAL_SECONDS → 2
✅Zero setup: needs no SendGrid, no Google, and no extra dependency — it runs out of the box on a fresh clone.
📱localhost on a phone means the phone itself. For a real cross-device scan, use a LAN IP or your public URL in APP_BASE_URL.

Security Vulnerability Lab Feature 05 / 05 · Authenticator App (TOTP)

Something you have — a code that rotates every 30 seconds.

At enrollment the server and your authenticator app agree on one shared secret. At login a correct password isn't enough — you must enter the 6-digit code the app derives from that secret and the current time. No SendGrid, no Google, works offline.

What happens

The authenticator (TOTP) flow

A one-time enrollment, then a time-based code at every login — and the shared secret behind it.

  1. 1

    User enrolls the authenticator

    From their profile they enable TOTP. The server generates a random 160-bit secret and shows it as a QR code (an otpauth:// link carrying the issuer label, 6 digits, and a 30s period).

  2. 2

    The app scans & stores the secret

    Google Authenticator or Authy saves it. Now both sides hold the same secret — and it never travels over the network again.

    🔑 shared secret — symmetric, held by both
  3. 3

    At login, password first

    The user enters username and password as usual.

  4. 4

    A correct password triggers a TOTP challenge

    Instead of logging in, the server asks for the current 6-digit code — the second factor.

    🔒 second factor — something you have
  5. 5

    User enters the rotating code

    The app computes it locally as HMAC(secret, time-step) — no internet needed — and it changes every 30 seconds.

    ⏱ rotates every 30s
  6. ✓

    Server checks the code against secret + time

    It recomputes the expected code from the shared secret and current time-step, allowing ±1 step of clock drift. Match → login completes.

    ● second factor cleared
What you do

Run it yourself

Like QR login, it works on a fresh clone. You just need a phone app to test, plus optional tuning.

  1. Nothing required — needs no SendGrid and no Google; works out of the box.
  2. Install an authenticator app (Google Authenticator, Authy, …) to actually test it.
  3. Enroll from your profile, scan the QR, then log in and enter a code.
  4. Optionally tune the label and drift tolerance ↓
.envall optional
# defaults shown — these only tune the label & drift
TOTP_ISSUER=Security Vulnerability Lab
TOTP_PERIOD_SECONDS=30
TOTP_SKEW_STEPS=1
Service → Environmentoptional tuning
# code length (6) and secret size (160-bit) are fixed
TOTP_ISSUER → Security Vulnerability Lab
TOTP_PERIOD_SECONDS → 30
TOTP_SKEW_STEPS → 1
✅Zero setup & offline: codes are computed locally on both sides from the shared secret, so no SMTP, no Google, and no network call to the phone.
⏱Clock drift: if a phone's clock is off, codes can fail. TOTP_SKEW_STEPS allows ±1 time-step (≈±30s) of tolerance.
